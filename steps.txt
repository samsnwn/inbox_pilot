The big picture (what your system really is)

Inbox Pilot is a pipeline:

Ingest an email (from simulator or Gmail later)

Store it safely (no duplicates)

Put work in a queue (so API stays fast)

Worker processes it:

classify + extract fields (AI)

retrieve relevant policies (RAG)

draft reply (AI)

Store results and show them via API

This is basically how real companies build: ingestion → async processing → results API.

Step-by-step roadmap (and what you’ll learn at each step)
Step 0 — Define “done” and constraints (scope control)

Goal: avoid building a monster.

We’ll keep UI optional (Swagger UI is enough)

Start with simulator ingestion

AI can be stubbed first, then swapped in

Deliverable: a short MVP checklist.

Key concept: scope boundaries.

Step 1 — Data model + DB migrations (the foundation)

Problem: if your DB model is messy, everything becomes painful.
We’ll design the tables so they match the pipeline:

emails

email_analysis

draft_replies

kb_documents

Key concept: data ownership and “single source of truth”.

Things you’ll understand:

Why analysis/draft are separate tables (versioning, reprocessing, auditing)

Why we store provider_message_id for idempotency

Deliverable: migration + tables created.

Step 2 — Ingestion endpoint + idempotency

Problem: email providers/webhooks can send duplicates. Networks retry. Clients retry.
So we must accept the same message multiple times without creating duplicates.

We’ll implement:

POST /emails/ingest

unique constraint: (provider, provider_message_id)

response indicates if it was idempotent

Key concept: idempotency and “at least once delivery”.

Deliverable: you can call ingest twice and still only have one email row.

Step 3 — Queue + worker (make it asynchronous)

Problem: AI calls can be slow and fail. You can’t block HTTP for that.
So we push heavy work to a worker.

We’ll implement:

Redis queue

a worker process

a job that receives email_id

Key concept: async processing, retries, and failure isolation.

Deliverable: ingest returns immediately, processing happens in background.

Step 4 — Pipeline: status transitions + error handling

Problem: you need to know what’s happening (new → processing → done/failed).
We’ll implement:

status field updates

“failed” states

basic retry strategy (later: DLQ)

Key concept: state machine thinking + reliability.

Deliverable: you can see the email status change as the worker runs.

Step 5 — AI classification + extraction (structured output)

Problem: LLMs can be inconsistent unless you constrain output.
We’ll start with a stub and then swap to real LLM with strict JSON schema.

We’ll implement:

category + priority + entities (order_id etc.)

store in email_analysis

Key concept: “LLM as a component” with contracts (schemas).

Deliverable: analysis row appears; can reprocess to regenerate.

Step 6 — Knowledge Base + retrieval (RAG-lite)

Problem: drafts should follow company policy and not hallucinate.
We add:

endpoints to store policy docs

retrieval that returns top relevant snippets

Start simple (keyword match), then upgrade to embeddings later.

Key concept: grounding and why RAG exists.

Deliverable: system pulls relevant KB snippets for an email.

Step 7 — Draft generation (grounded + citations)

Problem: you must be able to explain “why the draft says that”.
We’ll:

generate a reply draft using snippets

store “citations” (which docs were used)

Key concept: traceability and explainability.

Deliverable: draft reply + citations show up in /emails/{id}.

Step 8 — Testing and “interview polish”

We’ll add:

integration tests (ingest, process, reprocess)

basic metrics/logging

clear README + architecture diagram

Key concept: professional finish.
